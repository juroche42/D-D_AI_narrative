# US-06-04 — Démarrage de partie + Introduction narrative streamée

## Contexte

Quand le host démarre la partie, un `GameState` est créé en base, tous les joueurs sont redirigés vers `/game/:code`, et l'IA génère une introduction narrative immersive via RAG + OpenAI streaming. Le texte est diffusé token par token à tous les joueurs avec un effet machine à écrire **identique et simultané**, peu importe le nombre de joueurs connectés.

---

## Flux complet — du clic "Démarrer" jusqu'au texte affiché

### 1. Le host clique "Démarrer"

`RoomLobby.tsx` → `handleStart()` → `startGameAction(roomCode)` (Server Action)

`startGameAction` fait deux choses en séquence :
1. `updateRoomStatus(roomCode, userId, 'IN_PROGRESS')` — passe le salon en `IN_PROGRESS` en base et broadcast `room_status_changed` via SSE à tous les joueurs connectés au lobby.
2. `prisma.gameState.upsert({ where: { roomId }, create: { roomId } })` — crée le `GameState` lié à ce salon (idempotent : si appelé deux fois, met juste à jour `lastActivityAt`).

**Pré-requis vérifiés dans `updateRoomStatus`** :
- Au moins 2 joueurs dans le salon → sinon 422
- `Room.campaignId != null` (une campagne doit être sélectionnée) → sinon 422
- Seul le host peut déclencher → sinon 403

### 2. Redirection de tous les joueurs

- **Host** : `handleStart()` appelle `router.push('/game/${room.code}')` directement après que `startGameAction` répond `success: true`.
- **Autres joueurs** : `useRoomPlayers` reçoit l'event SSE `room_status_changed` avec `status: 'IN_PROGRESS'`. Un `useEffect` dans `RoomLobby` surveille `roomStatus` et appelle `router.push('/game/${roomCode}')` dès qu'il vaut `'IN_PROGRESS'`. Tous les joueurs sont donc redirigés simultanément, quelques millisecondes après le host.

### 3. Page de jeu (Server Component)

`app/(game)/game/[code]/page.tsx` — exécuté côté serveur à l'arrivée sur la page.

Vérifications :
- Session active → sinon redirect `/login`
- `Room.status === 'IN_PROGRESS'` → sinon redirect `/lobby`
- Utilisateur membre du salon → sinon redirect `/lobby`

Si tout est ok, rend le composant client `<GameView roomCode={code} roomName={room.name} />`.

### 4. GameView déclenche le stream

`components/game/GameView.tsx` — composant client (`'use client'`).

Au premier rendu, un `useEffect` appelle `startStream(roomCode)` depuis le hook `useNarrativeStream`. Cela déclenche un `fetch` vers `GET /api/game/:code/stream?type=intro`.

**Problème React 18 Strict Mode** : en développement, React double-invoque les `useEffect` (mount → unmount → mount) pour détecter les effets de bord. Sans précaution, `startStream` serait appelé deux fois simultanément, créant deux streams parallèles dont les tokens s'intercaleraient dans le même state → texte illisible.

**Fix** : `useNarrativeStream` utilise un `AbortController`. Chaque appel à `startStream` commence par **aborter le fetch précédent** s'il y en a un. Le premier fetch est donc annulé avant de recevoir des données, le second prend le relais proprement. L'`AbortError` est ignoré silencieusement.

### 5. Route SSE — trois cas selon l'état du serveur

`app/api/game/[code]/stream/route.ts`

Après vérification auth + membership, la route détermine quel cas s'applique :

---

#### Cas 1 — Texte déjà persisté en base (rechargement de page)

`GameState.narrativeContext` est non vide → le texte a déjà été généré lors d'une session précédente.

La route ne rappelle **pas** OpenAI. Elle relit le texte depuis la base et le renvoie en chunks de 10 caractères avec un délai de 10ms entre chaque :

```
chunk 1 : "Sous un c"  →  10ms  →  chunk 2 : "iel bleu "  →  10ms  →  ...
```

Résultat : l'effet machine à écrire est reproduit depuis le cache, à une vitesse homogène (~100 chars/sec), sans aucun coût API supplémentaire.

---

#### Cas 2 — Première connexion, personne ne génère encore

`narrativeContext` est vide **et** aucun verrou n'est pris pour cette room → ce joueur est le premier à arriver.

La route :
1. Acquiert le **verrou** (`acquireLock(roomCode) → true`) pour bloquer les autres.
2. Initialise le **buffer mémoire** (`initBuffer(roomCode)`) — une structure `{ tokens: string[], done: boolean }` stockée dans `globalThis.__tokenBuffers`.
3. Appelle `generateCampaignIntroduction(ctx, onChunk)`.
4. À chaque token reçu d'OpenAI via `completeStream`, `onChunk` fait deux choses simultanément :
   - `pushToken(roomCode, token)` — ajoute le token au buffer mémoire.
   - `send({ type: 'token', token })` — envoie le token à ce joueur via SSE.
5. Une fois terminé : `markGenerationDone(roomCode)` marque le buffer comme complet, `releaseLock(roomCode)` libère le verrou.

`generateCampaignIntroduction` s'occupe en parallèle de persister le texte complet en `NarrativeEntry` et dans `GameState.narrativeContext` via `onDone`.

---

#### Cas 3 — Connexion simultanée, génération déjà en cours

`narrativeContext` est vide **et** le buffer mémoire existe déjà (`isBuffered(roomCode) → true`) ou le verrou est pris.

Ce joueur ne peut pas générer (il serait trop tard et créerait un texte différent). Il surveille le **buffer mémoire partagé** :

```ts
let offset = 0;
while (Date.now() < deadline) {
  const { tokens, done } = getTokensFrom(roomCode, offset);
  for (const token of tokens) {
    send({ type: 'token', token }); // relay immédiat
    offset++;
  }
  if (done && tokens.length === 0) break;
  await new Promise((r) => setTimeout(r, 50)); // poll toutes les 50ms
}
```

Concrètement : toutes les 50ms, il lit les nouveaux tokens que le générateur a ajoutés depuis son dernier poll, et les envoie à son propre client SSE. Le décalage maximum entre le joueur 1 et les autres est de **50ms** — imperceptible.

**Résultat** : tous les joueurs voient exactement le même texte apparaître avec le même effet typewriter, quasiment en simultané.

---

### 6. Hook client `useNarrativeStream`

`hooks/useNarrativeStream.ts` — lit la réponse SSE du serveur côté client.

La réponse HTTP est un stream continu. Le hook lit le body via `ReadableStreamDefaultReader`, accumule les données dans un buffer string, découpe sur les sauts de ligne, et parse chaque ligne `data: {...}` comme un event JSON.

Pour chaque event reçu :
- `{ type: 'start' }` → ignoré (juste un signal d'ouverture)
- `{ type: 'token', token: '...' }` → `setState(prev => ({ ...prev, text: prev.text + token }))` — concatène le token au texte affiché
- `{ type: 'done' }` → `setState(prev => ({ ...prev, status: 'done' }))` — cache le curseur
- `{ type: 'error', error: '...' }` → affiche l'erreur

`status` peut valoir : `'idle'` | `'streaming'` | `'done'` | `'error'`

### 7. Affichage dans GameView

`components/game/GameView.tsx` affiche `text` dans un `<p>` avec `whitespace-pre-wrap`.

Pendant `status === 'streaming'`, un `<span>` avec `animate-pulse` simule un curseur clignotant après le dernier caractère. Quand `status === 'done'`, le curseur disparaît et une ligne "— Fin de l'introduction —" apparaît.

## Architecture du buffer de tokens (generationLock.ts)

```
GlobalThis.__generatingRooms  →  Set<roomCode>                   (verrou simple)
GlobalThis.__tokenBuffers      →  Map<roomCode, { tokens, done }> (buffer de tokens)
```

Ces deux structures sont stockées sur `globalThis` (même pattern que `sseManager` pour les clients SSE du lobby). Cela garantit qu'une seule instance partagée existe entre toutes les requêtes Next.js dans le même processus Node.js.

**Cycle de vie d'un buffer** :
1. `initBuffer` → créé vide lors de l'acquisition du verrou
2. `pushToken` → alimenté à chaque chunk OpenAI (appelé par le générateur)
3. `markGenerationDone` → marqué `done: true` quand OpenAI termine
4. Auto-nettoyage après 60s → supprimé de la Map (libère la mémoire)

**Limite** : ce système fonctionne pour une instance Next.js unique. En production multi-instance (plusieurs pods), il faudrait Redis Pub/Sub à la place.

## Modèles Prisma

```prisma
enum NarrativeEntryType { NARRATION ACTION DICE_ROLL SYSTEM }

model GameState {
  id               String           @id @default(cuid())
  roomId           String           @unique
  currentTurn      Int              @default(1)
  worldState       Json             @default("{}")
  narrativeContext String           @default("")   // texte complet de l'intro, sert de cache
  lastActivityAt   DateTime         @default(now())
  narrativeEntries NarrativeEntry[]
  @@map("game_states")
}

model NarrativeEntry {
  id          String             @id @default(cuid())
  gameStateId String
  turn        Int
  type        NarrativeEntryType  // NARRATION pour l'intro
  content     String
  authorId    String?             // null pour les narrations du DM IA
  @@index([gameStateId, turn])
  @@map("narrative_entries")
}
```

`GameState.narrativeContext` joue un double rôle : cache pour éviter de rappeler OpenAI, et contexte pour les futures générations (le DM IA s'en souviendra au tour 2, 3...).

`NarrativeEntry` stocke chaque message de la partie : narrations du DM, actions des joueurs, jets de dés, etc. Indexé par `(gameStateId, turn)` pour récupérer efficacement l'historique d'un tour.

## API

### `GET /api/game/:code/stream?type=intro`

**Auth** : requise — membre du salon
**Réponses HTTP** :
- `401` — non authentifié
- `403` — non membre du salon
- `404` — salon introuvable
- `409` — salon non `IN_PROGRESS`
- `200` — `text/event-stream` (stream ouvert)

**Events SSE émis sur le stream** :

| Event | Payload | Quand |
|-------|---------|-------|
| `start` | `{ type: 'start' }` | Dès l'ouverture du stream |
| `token` | `{ type: 'token', token: '...' }` | À chaque fragment de texte |
| `done` | `{ type: 'done' }` | Fin du texte |
| `error` | `{ type: 'error', error: '...' }` | En cas d'erreur OpenAI ou timeout |

---

## Sécurité

- `Campaign.systemPrompt` est utilisé uniquement côté serveur dans `buildIntroSystemPrompt`. Il n'est **jamais** inclus dans les events SSE ni dans aucune réponse API.
- Vérification de membership à chaque ouverture du stream (un joueur expulsé ne peut pas accéder au stream).
- Le `GameState` est créé via `upsert` — idempotent si `startGameAction` est appelé plusieurs fois.
- Le buffer mémoire est nettoyé automatiquement après 60 secondes.
