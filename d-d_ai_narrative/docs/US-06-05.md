# US-06-05 — Tours de jeu : actions suggérées et narration de scène

## Résumé

Après l'introduction générée en US-06-04, la partie entre dans une boucle de tours :

1. L'IA génère 3 actions suggérées cohérentes avec la scène courante
2. Les joueurs sélectionnent une action (ou saisissent une action libre)
3. L'IA génère la scène suivante en streaming, basée sur l'action choisie et l'historique
4. Le cycle recommence pour le tour suivant

Les votes réels (broadcast SSE, compteurs, timer automatique) sont implémentés en US-07.
Cette US pose les fondations : IDs d'actions persistées, placeholders "0 votes", timer statique.

---

## Architecture multi-phase côté client

```
intro_loading  ──→  actions_loading  ──→  voting  ──→  scene_loading
     ↑                                                        │
     └──────────────── (retour tour suivant) ←────────────────┘
```

### Phases

| Phase | Description |
|-------|-------------|
| `intro_loading` | Stream SSE `type=intro` — introduction narrative token par token |
| `actions_loading` | Requête SSE `type=actions` — génération des 3 actions (non-streaming) |
| `voting` | Joueurs sélectionnent une action ou saisissent une action libre |
| `scene_loading` | Stream SSE `type=scene` — narration de la scène suivante token par token |

---

## Contrat API SSE étendu

### `GET /api/game/:code/stream?type=actions`

Génère et retourne les 3 actions suggérées pour le tour courant. Non-streaming — la réponse SSE contient un seul événement `actions`.

**Flux d'événements :**
```
data: {"type":"start"}
data: {"type":"actions","actions":[{"id":"cuid1","content":"Inspecter les murs","type":"SUGGESTED"},...],"turn":1}
data: {"type":"done"}
```

**Erreur si aucune narration trouvée :**
```
data: {"type":"error","error":"Aucune narration trouvée pour ce tour"}
```

### `GET /api/game/:code/stream?type=scene&actionId=<id>`
### `GET /api/game/:code/stream?type=scene&action=<texte%20encodé>`

Génère la scène narrative suivante en streaming. Accepte soit l'ID d'une `TurnAction` existante, soit un texte libre encodé en URL.

**Flux d'événements :**
```
data: {"type":"start","turn":1,"action":"Inspecter les murs"}
data: {"type":"token","token":"Vous approchez"}
data: {"type":"token","token":" prudemment..."}
... (tokens successifs)
data: {"type":"done"}
```

## Modèle de données

### `TurnAction`
Stocke les actions proposées par l'IA (SUGGESTED) ou saisies librement (FREE).

```
TurnAction {
  id          String         @id
  gameStateId String         → GameState (cascade delete)
  turn        Int            // numéro de tour
  type        SUGGESTED|FREE
  content     String         // texte de l'action
  authorId    String?        // null = IA, userId = joueur
  votes       Vote[]         // relation US-07 (placeholder)
}
```

### `Vote` (stub US-07)
Modèle minimal pour satisfaire la relation `TurnAction.votes`. Enrichi en US-07 avec la logique de vote.

```
Vote {
  id          String    @id
  gameStateId String    → GameState (cascade delete)
  userId      String    → User
  actionId    String    → TurnAction (cascade delete)
  turn        Int
  @@unique([gameStateId, userId, turn])  // 1 vote par joueur par tour
}
```

## Historique conversationnel

`generateTurnActions` et `generateSceneNarrative` chargent les dernières `NarrativeEntry` pour construire un contexte conversationnel cohérent :

- **`NARRATION`** → `role: 'assistant'` (narrations précédentes de l'IA)
- **`ACTION`** → `role: 'user'` avec préfixe `[Système]` (actions collectives résolues)

**Limite :** 8 entrées pour `generateTurnActions`, 10 pour `generateSceneNarrative` — pour rester dans la fenêtre de contexte OpenAI (~2000 tokens max).

---

## Fonctions narrativeService.ts

### `generateTurnActions(ctx, lastNarration) → string[]`

- Charge l'historique (8 entrées)
- Appelle `complete()` (non-streaming) avec consigne JSON stricte
- Parse `{"actions": ["a1","a2","a3"]}` avec fallback 3 actions génériques
- Retourne exactement 3 chaînes

### `saveTurnActions(gameStateId, turn, actions) → {id, content, type}[]`

- Supprime les SUGGESTED existantes pour ce tour (idempotent)
- Crée 3 nouvelles `TurnAction` en parallèle
- Retourne les objets avec leurs IDs (nécessaires au vote)

### `generateSceneNarrative(ctx, chosenAction, chosenActionId, onChunk) → string`

1. Charge historique (10 entrées)
2. RAG sur `"${action} ${startLocation}"`
3. Persiste `NarrativeEntry(ACTION)` avant le stream
4. Stream `completeStream()` token par token via `onChunk`
5. Persiste `NarrativeEntry(NARRATION)` après stream
6. `gameState.update({ currentTurn: { increment: 1 }, narrativeContext: résumé })`
7. Retourne `fullContent`

## Hook `useNarrativeStream`

Refactorisé pour être multi-type. Signature :

```typescript
useNarrativeStream(roomCode: string, type: 'intro' | 'actions' | 'scene')
```

`startStream(customUrl?: string)` — sans `customUrl` utilise l'URL par défaut pour le type. Avec `customUrl`, permet de passer `actionId` ou `action` pour `type=scene`.

Nouveaux champs state :
- `actions: TurnActionItem[]` — populés par l'événement SSE `type=actions`
- `currentTurn: number` — mis à jour par les événements `start` et `actions`

## Persistance narrative par tour

Après chaque tour joué, la base de données contient :

```
NarrativeEntry(turn=N, type=ACTION,    content="texte action choisie")
NarrativeEntry(turn=N, type=NARRATION, content="narration générée")
```

`GameState.currentTurn` est incrémenté de 1.
`GameState.narrativeContext` est mis à jour avec un résumé `[Tour N] Action : "..." → narration...`.
