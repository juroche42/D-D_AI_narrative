# US-02-04 — Navbar persistante + déconnexion utilisateur

## Vue d'ensemble

Cette US implémente la navbar persistante du projet D&D AI Narrative avec affichage conditionnel selon l'état d'authentification, ainsi que la fonctionnalité de déconnexion (logout) via NextAuth v5.

---

## Architecture

```
components/
└── layout/
    ├── Navbar.tsx          ← Server Component — lit la session via auth()
    ├── NavbarClient.tsx    ← Client Component — menu mobile + liens nav
    ├── UserMenu.tsx        ← Client Component — dropdown avatar connecté
    ├── LogoutButton.tsx    ← Client Component — bouton déconnexion (SRP)
    └── index.ts            ← Barrel export

app/
└── (auth)/
    └── logout/
        └── actions.ts      ← Server Action : logoutAction()
```

## Composants détaillés

### 1. Navbar.tsx — Server Component

```typescript
export async function Navbar() {
  const session = await auth();
  const user = session?.user ?? null;
  // ...
}
```

**Pourquoi un Server Component ?**
Avec `useSession()` côté client, la navbar s'affiche d'abord en mode "non connecté" puis switch après le fetch de session — provoquant un flash visible (**FOUC** : Flash Of Unauthenticated Content). Avec `auth()` côté serveur, l'état est correct dès le premier rendu HTML.

La session est lue **une seule fois** côté serveur puis passée en cascade via props — pas de double fetch, pas de round-trip réseau supplémentaire.

### 2. NavbarClient.tsx — Client Component

Reçoit `user: { id, username } | null` en prop depuis `Navbar`. Gère :
- Liens de navigation desktop
- Menu mobile (burger/fermer)
- Affichage conditionnel : `UserMenu` si connecté, bouton "Connexion" sinon

### 3. UserMenu.tsx — Client Component

Dropdown shadcn (`DropdownMenu`) affiché quand l'utilisateur est connecté :
- Trigger : pill avatar avec username + icône user
- Items : lien "Mon Profil", séparateur, bouton logout

> **NOTE future** : quand la ProfilePage sera livrée, le `LogoutButton` devra être déplacé dans `/profile` et retiré du `UserMenu`. Retirer `<LogoutButton />` dans `UserMenu.tsx` sans toucher aux autres fichiers (SRP respecté).

### 4. LogoutButton.tsx — Client Component

Composant à responsabilité unique : déclencher la déconnexion. Deux variantes :
- `variant="header"` : bouton compact avec icône, texte "Quitter" masqué sur mobile
- `variant="menu-item"` : item pleine largeur dans le menu dropdown/mobile

Utilise `useTransition` pour afficher un spinner pendant la déconnexion sans bloquer l'UI.

### 5. logoutAction() — Server Action

```typescript
'use server';
export async function logoutAction(): Promise<void> {
  await signOut({ redirectTo: '/' });
}
```

**Pourquoi une Server Action et pas un fetch client ?**
`signOut()` de NextAuth v5 doit être appelé **côté serveur** pour que le cookie de session soit invalidé avant la redirection. Un `fetch('/api/auth/signout')` côté client ne garantit pas l'ordre des opérations et peut laisser des états incohérents.

## Flux d'authentification — Navbar

### Non connecté
```
Navbar (Server) → auth() → null
→ NavbarClient reçoit user=null
→ Affiche bouton "Connexion" (link /login)
```

### Connecté
```
Navbar (Server) → auth() → { user: { id, username } }
→ NavbarClient reçoit user={ id, username }
→ Affiche UserMenu (avatar pill + dropdown)
  └── Dropdown : Mon Profil | Se déconnecter
```

### Déconnexion
```
1. Clic sur "Se déconnecter" dans UserMenu ou menu mobile
2. LogoutButton → useTransition → logoutAction() [Server Action]
3. signOut({ redirectTo: '/' }) invalide le cookie JWT côté serveur
4. Redirection vers /
5. Navbar repasse en mode non connecté (session null)
```

## Séparation Server / Client — Pourquoi ?

| Composant | Type | Raison |
|-----------|------|--------|
| `Navbar` | Server | Appelle `auth()` — impossible dans un Client Component |
| `NavbarClient` | Client | `useState` pour le menu mobile |
| `UserMenu` | Client | `DropdownMenu` radix UI (état ouvert/fermé) |
| `LogoutButton` | Client | `useTransition` pour le spinner de chargement |

**Règle critique** : `Navbar.tsx` n'a **aucun** `'use client'`. Les données de session sont passées en props vers les composants clients — jamais re-fetchées côté client.

## app/layout.tsx — Intégration

```
RootLayout
└── <Providers>          ← SessionProvider (client)
    ├── <Navbar />        ← Server Component (appelle auth())
    ├── <main>            ← Contenu des pages
    └── <footer>          ← Pied de page
```

**Important** : `Navbar` est à l'intérieur de `<Providers>` pour que le `SessionProvider` soit disponible (requis par NextAuth v5 pour les composants clients qui utilisent `useSession()`).
